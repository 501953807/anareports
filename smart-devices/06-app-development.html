第六章：APP功能规划与开发团队配置

APP MVP版本(3-4个月)40-70万元，标准版(5-7个月)70-120万元，完整版(8-12个月)120-200万元。6-7人团队月成本10-14万。技术栈：Go/Java后端 + Swift/Kotlin双端 + Vue/React管理后台。

6.1 APP功能模块规划

模块功能优先级开发周期技术要点深度解读
设备配对BLE配对手环 + WiFi配药盒P0 必须2-3周BLE GATT协议 / WiFi SmartConfig配对成功率是留存第一杀手——竞品平均配对失败率约8%(老人操作能力弱导致)。需设计"一键配对"流程，减少用户操作步骤。
实时定位地图高德/百度地图API集成，实时轨迹回放P0 必须3-4周地图SDK + GCJ-02转换 + 轨迹插值高德地图API免费额度5,000次/天/KEY，超出后¥0.01/次。1,000台设备×每天10次查询=10,000次/天，需申请企业级配额或自建缓存层。
电子围栏多边形围栏设置、出入告警推送P0 必须2-3周地理围栏API + 多边形点包含算法多边形围栏算法(Ray Casting)计算复杂度O(n)，n为多边形顶点数。建议限制用户最多设置10个顶点，避免性能问题。圆形围栏作为降级方案。
心率/血氧监测PPG数据可视化、异常阈值预警P0 必须2-3周时序数据图表 + 阈值规则引擎心率阈值设定需个性化——静息心率60-100bpm是"医学标准"，但老人长期静息心率50bpm属正常。建议用"基线偏差法"(偏离个人基线±20%才告警)。
跌倒检测告警IMU数据触发SOS推送P0 必须2-3周端侧AI模型 + 告警优先级路由跌倒检测F1分数行业标杆是Apple Watch(>0.95)，但需要持续训练。建议初期用开源模型(MicroPython TinyML)，后期收集真实数据微调。
SOS紧急呼救一键拨号给预设联系人P0 必须1-2周Telephony API + 通讯录权限SOS是"信任锚"——一次误报(假跌倒)可能导致用户关闭功能。建议SOS触发前增加"长按3秒确认"防误触机制，同时保留自动检测SOS。
用药提醒/药盒联动定时提醒、服药记录、家属通知P1 重要2-3周本地通知 + 云端调度 + BLE指令用药依从性是全球性问题——WHO统计慢性病患者用药依从性仅约50%。APP提醒+药盒硬件双重保障可提升至70-80%，但仍不足。
家属多人共享多账号绑定同一老人设备P1 重要2-3周设备-用户多对多 + 权限管理多人共享带来"责任分散"效应——每个家属都以为别人在关注，结果没人关注。建议设置"主要联系人"(唯一SOS接收者)和"次要联系人"(仅查看)。
健康周报/月报数据聚合生成可视化报告P2 增值2-3周数据聚合 + PDF/图片生成健康周报是"粘性剂"——有周报功能的APP用户7日留存高15-20%。建议加入"与上月对比"和"医生建议"增强价值感。
后台管理Web端运营人员管理后台P2 增值3-4周Vue/React + ECharts + RBAC运营后台在MVP阶段可用Google Sheets替代(通过API写入)。节省3-4周开发时间和¥5-10万成本，等日活>1,000再自建。

推送APNs(iOS) + 厂商通道(Android) + 微信订阅消息 + 阿里云SMS(紧急兜底)推送通道覆盖率：APNs≈99%(iOS) + 厂商通道≈95%(Android) + 微信≈80%。四通道叠加覆盖约99.5%。短信兜底成本约¥0.04-0.06/条。

小程序版微信原生 / uni-app，仅提醒/查看功能，作为轻量入口(10-20万，1-2个月)小程序是获客利器——无需下载、微信内分享。但小程序BLE能力受限(iOS不可用后台BLE)。建议小程序做"轻入口"，完整功能引导下载APP。
细微信号：微信生态正在成为IoT入口新战场
2024年微信小程序IoT设备连接数突破5亿，微信开放了BLE后台扫描能力(iOS 17.4+/Android 12+)。这意味着"手环+药盒"可以完全在微信生态内运行，无需独立APP。降低获客成本约¥50-80/用户(免去了APP下载转化漏斗的80%流失)。
五大学派APP开发分析
凯恩斯主义 ：APP是数字基础设施的一部分，政府应推动"适老化数字服务标准"强制落地。建议参与工信部"互联网应用适老化通用设计规范"团体标准制定，将APP适老化纳入行业标准。
奥地利学派 ：去中心化趋势明显——小程序/IoT设备本身即是"APP"。独立APP的必要性在下降。选择微信生态而非独立APP，是对市场自发秩序的顺应，而非妥协。
后凯恩斯主义(Minsky) ：开发团队的人力成本是"固定支出"，而订阅收入是"可变收入"。在收入不确定时维持6-7人团队构成Minsky式脆弱性。建议用外包+兼职测试降低固定成本，等PMF验证后再扩编。
新制度经济学 ：平台即制度——微信生态是一套"治理规则"。在微信内开发意味着接受微信的规则(审核/抽成/数据归属)。独立APP是完全的内部制度。选择取决于"交易成本"比较。
行为经济学 ："默认效应"解释了为什么微信生态获客成本低——老人已经在使用微信，不需要改变习惯。独立APP需要"下载-注册-配对"三步改变，每一步都有约20-30%流失。损失厌恶：老人害怕"弄坏手机"，不愿尝试新APP。
案例深挖：小米运动APP的"功能膨胀"教训 — 少即是多
问题小米运动APP功能过多(运动记录+社交+商城+资讯)，导致核心功能(心率/定位)入口深、操作路径长
数据尼尔森十法则评分：状态可见性4/5，操作效率3/5(查看心率需3步)，系统与现实匹配2/5(大量医学术语)
我们的对策首页只做三件事：①今日位置(大地图) ②今日心率(大数字) ③今日用药(大按钮)。其他功能全部折叠在"更多"二级页面中
字体策略基础字号18sp(安卓标准16sp的1.125倍)，关键数据字号24sp。按钮最小触控面积48×48dp(安卓规范)
色彩策略对比度≥4.5:1(WCAG AA级)。避免红绿配色(色盲老人无法区分)。用"绿色=正常""红色=异常"的通用语义
警示不要做"功能竞赛"——Apple Watch功能最多但老人使用率最低。功能≠价值，简洁=价值。MVP版本功能不超过8个

实操：APP开发完整教程 — 从环境搭建到上线发布

第一步：Flutter项目初始化(30分钟搞定)

  1. 下载Flutter SDK: 访问 flutter.dev/downloads，下载macOS版flutter_macos_v3.24.zip，解压到~/development/flutter。
  2. 配置环境变量: 在~/.zshrc中添加export PATH="$PATH:~/development/flutter/bin"，然后source ~/.zshrc。
  3. 运行flutter doctor：检查Android SDK、Xcode、Dart插件(VS Code)是否就绪。
  4. 创建项目: flutter create wellness_watch && cd wellness_watch。
  5. 加入核心依赖(pubspec.yaml): ble_platform_detection: ^1.0、flutter_blue_plus: ^1.33、http: ^1.2、shared_preferences: ^2.2、geolocator: ^12.0、amap_map_fluttify: ^0.0.14。

第四步：推送通知多渠道配置

  1. APNs(iOS)：在Apple Developer Portal创建Push Notification Certificate(.p12)，上传到阿里云推送控制台。
  2. 厂商通道(Android)：注册小米推送(XPS)、华为推送(HPS)、OPPO Push、vivo Push四个厂商通道。一个APP只需配一次，可覆盖95% Android设备。
  3. 微信模板消息：在微信公众平台(mp.weixin.qq.com)注册服务号，配置"用药提醒"和"SOS告警"两个模板。用户授权后通过微信推送。
  4. 短信兜底：接入阿里云短信服务(aliyun.com/sms)，P0级告警(跌倒/越界)同时触发APP推送+短信。

第七步：APP灰度发布策略(先1%后100%)

  1. 内部测试版：通过Firebase Test Lab或Google Play Internal Testing，先对内部分发，2人开发团队连续使用3天。
  2. Alpha灰度(1%)：在App Store设为"TestFlight"测试，限定邀请码。收集Crashlytics崩溃率，目标<0.5%。
  3. Beta灰度(10%)：华为/小米应用市场设"先锋体验官"计划，用户报名后优先安装。收集用户反馈问卷(用Typeform，¥299/月)。
  4. 全量发布：各项指标达标后全量上架。发布后24小时内密切监控崩溃率和闪退率。
