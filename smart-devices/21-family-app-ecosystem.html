第二十一章：家庭APP生态 — 多用户/照护者面板/告警通知

手环的用户是老人，但APP的实际使用者是子女和照护者。"买的人不用，用的不买单"是典型三角色分离。本章节设计多账户权限管理和照护者仪表盘。

21.1 三角色权限体系

角色权限数据可见性操作能力深度解读
老人(被照护者)只读设备自身数据心率/血压/位置/药盒状态SOS按钮/吃药确认/音量调节角色权限最小化原则：老人仅有"被观察"权限而无"管理"权限，这反映了照护关系中的权力不对等——子女掌握数据控制权，老人仅有知情权
子女(主要照护者)完整管理权限全部健康+位置+用药数据围栏设置/服药提醒/紧急联系人/消息通知子女承担"数据经纪人"角色，其管理权限本质上是代理决策权。但当多个子女共享admin权限时，需引入权限冲突解决机制(如投票制)
社区医护(协作方)只读健康数据心率/血压/睡眠趋势发送健康建议/标注异常医护角色的"只读+建议"权限设计体现了"数据赋能但不越权"的原则，避免数字医疗侵入家庭自主决策空间

21.2 多账户家庭成员架构

功能设计细节技术实现深度解读
一机多人绑定一个手环最多绑定5个子女账号device_id → user_id映射表, Redis缓存5人上限基于 Dunbar 数字理论——人类稳定社交圈约150人，核心照护圈约5人。超过5人会导致责任分散效应(diffusion of responsibility)
主副照护者1个主照护者(决策) + 4个副照护者(知情)role字段: admin/collaborator/viewer主照护者设定避免了"多头指挥"问题——当多个子女同时发出矛盾指令时，系统需要明确的优先级仲裁机制
消息分级P0(紧急): 全员推送 + 电话语音
P1(重要): 仅admin通知
P2(普通): 下次登录查看		Firebase Cloud Messaging + 短信网关消息分级本质上是"注意力资源配置"问题——P0级别消耗全家注意力资源，P2级别仅消耗admin注意力，防止信息过载导致的告警疲劳
轮换照顾工作日子女A / 周末子女B 轮值time-slot角色分配器轮值机制反映了家庭照护的"隐性劳动分工"问题——实际照护负担往往不成比例地落在某一子女身上，系统化的轮值有助于公平分担

21.3 告警通知规则引擎

告警类型触发条件通知对象通知方式升级策略深度解读
SOS手动长按3秒所有紧急联系人Push+电话+短信5分钟无确认→通知社区/120SOS是"主动求救"信号，三重通道覆盖确保信息可达性。5分钟升级窗口基于黄金救援时间理论——心脏骤停后5分钟内干预存活率最高
跌倒检测加速度+心率骤变主要照护者Push+电话10分钟无响应→群发其他家人10分钟窗口比SOS长一倍，因为跌倒检测存在假阳性可能(约5%)，系统给予主要照护者初步核实的时间窗口
越界告警超出电子围栏500m主要照护者Push30分钟未归→电话确认30分钟是最长的非紧急升级窗口——老人外出购物/访友是常见场景，过早电话确认反而构成"数字监视"感受
心率异常HR>120或<50持续5分钟主要照护者Push持续异常→建议就医提示心率异常属于"渐进式风险"而非"突发事件"，系统从"通知"降级为"建议"，体现了风险等级与响应强度的正相关设计
药盒漏服约定时间后2小时未开门主要照护者Push超过4小时→电话确认2小时缓冲期考虑了老人记忆衰退导致的"服了但忘记确认"场景，4小时电话确认则覆盖了真正漏服的风险
设备低电量电量<10%所有绑定者Push24小时未充电→电话提醒低电量告警的特殊性在于它影响的是"监测能力"而非"老人安全"本身，因此24小时宽限期反映了系统对"设备维护"与"人身安全"的优先级区分

21.4 照护者仪表盘布局

区块内容刷新频率深度解读
顶部状态栏老人头像+姓名+当前状态(在线/离线/告警)+最近位置实时状态栏是整个APP的"注意力锚点"——用户打开APP的第一视线落点，决定了信息架构的优先级设计
健康卡片今日心率(曲线)+血压+SpO2+睡眠评分(简图)每5分钟5分钟刷新频率平衡了"数据新鲜度"与"电池/流量消耗"，同时避免了子女因过度关注实时数据而产生的"数字焦虑症"
用药卡片今日服药进度(✓/×/⏰)+下次服药时间+剩药量实时(药盒联动)用药依从性是老年慢病管理的最大痛点——全球范围内65岁以上老人用药依从性仅约50%，数字化提醒可将依从性提升至70-80%
位置卡片地图缩略图+位置名称+围栏状态每5分钟/移动触发位置数据是最敏感的隐私信息之一——"电子围栏"设计在安全监护与隐私侵犯之间划了一条微妙的界线，需获得老人明确同意
告警列表最近7天告警汇总(可点击查看详情)实时7天窗口是经过实证研究的最优长度——太短无法发现趋势(如跌倒频率上升)，太长则信息过载
周报/月报健康趋势总结+异常事件统计+医生建议每周一推送周期性报告将"碎片化数据"转化为"结构化洞察"，是数据价值从L1(L2)跃迁到L3的关键载体

老人手环产品核心表只有5张，遵循最小必要原则。以下是最关键的 users 表设计：
SQL: 核心表结构 
-- 用户表(子女+老人+医护三种角色)
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  role VARCHAR(20) CHECK (role IN ('caregiver','elderly','nurse')),
  phone VARCHAR(20) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

-- 老人档案表
CREATE TABLE profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id),
  device_id VARCHAR(64) UNIQUE,
  first_name VARCHAR(50),
  date_of_birth DATE,
  medical_conditions TEXT[],  -- 慢性病数组:['高血压','糖尿病']
  emergency_contacts JSONB DEFAULT '[]'::jsonb
);

-- 家庭成员关系表(一人多设备/一设备多 caregiver)
CREATE TABLE family_relations (
  id SERIAL PRIMARY KEY,
  family_id UUID NOT NULL,
  user_id UUID REFERENCES users(id),
  role VARCHAR(20) CHECK (role IN ('primary','collaborator','viewer')),
  invited_at TIMESTAMP DEFAULT NOW(),
  accepted_at TIMESTAMP
);

-- 健康数据时序表(Redis存热数据, PG存冷数据)
CREATE TABLE health_events (
  id BIGSERIAL PRIMARY KEY,
  device_id VARCHAR(64) NOT NULL,
  event_type VARCHAR(30),  -- 'heartbeat','fall','gps','medication'
  payload JSONB NOT NULL,
  recorded_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_device_event ON health_events(device_id, recorded_at DESC);
CREATE INDEX idx_event_type ON health_events(event_type) WHERE event_type = 'fall';

src/routes/auth.ts 
export const login = async (req, res) => {
  const { phone, password } = req.body;
  const user = await pool.query(
    'SELECT * FROM users WHERE phone=$1', [phone]);
  if (!user.rows[0]) return res.status(401).json({error:'用户不存在'});
  const valid = await bcrypt.compare(password, user.rows[0].password_hash);
  if (!valid) return res.status(401).json({error:'密码错误'});
  const token = jwt.sign({userId: user.rows[0].id, role: user.rows[0].role},
    process.env.JWT_SECRET, {expiresIn: '7d'});
  res.json({token, role: user.rows[0].role});
};

router.post('/telemetry', authenticateToken, async (req, res) => {
  const { deviceId, telemetry } = req.body;
  // 批量插入(高性能)
  const values = telemetry.map(t =>
    [`$1`, `$2`, JSON.stringify(t.payload), `NOW()`]);
  const query = `INSERT INTO health_events(device_id, event_type, payload, recorded_at) VALUES ${values.join(',')} ON CONFLICT DO NOTHING`;
  await pool.query(query, [`deviceId`, ...values.flat()]);
  // 同时写入 Redis 用于实时告警
  await redis.lpush(`device:${deviceId}:live`, JSON.stringify(telemetry));
  await redis.expire(`device:${deviceId}:live`, 86400);
  res.json({ok: true});
});

// 告警规则引擎(简化版)
const ALERT_RULES = [
  { type: 'fall', threshold: 1, windowMs: 5000,  notifyAll: true },
  { type: 'high_hr', threshold: 120, windowMs: 300000, notifyAll: false },
  { type: 'missed_med', threshold: 1, windowMs: 7200000, notifyAll: false },
];

// 检查最近一条事件是否触发告警
async function checkAlerts(deviceId, event) {
  const rule = ALERT_RULES.find(r => r.type === event.type);
  if (!rule) return;
  const recent = await redis.lrange(`device:${deviceId}:${event.type}`, 0, -1);
  if (recent.length >= rule.threshold) {
    // 触发告警 → 调用推送服务
    await sendPushNotification(deviceId, event.type);
    // 清理队列
    await redis.del(`device:${deviceId}:${event.type}`);
  }
}

FCM 推送分为三层：P0 紧急告警 同时走 FCM 高优先级 + 阿里云短信(容灾)、P1 重要通知 仅 FCM、P2 普通信息 存入 APP 内消息队列。

// services/fcm.ts (Node.js + Firebase Admin SDK)
import { initializeApp, cert } from 'firebase-admin/app';
import { getMessaging } from 'firebase-admin/messaging';

initializeApp({
  credential: cert({
    projectId: 'elderly-care-prod',
    privateKey: process.env.FCM_PRIVATE_KEY,  // 从密钥JSON读取
    clientEmail: process.env.FCM_CLIENT_EMAIL,
  })
});

const messaging = getMessaging();

export async function sendEmergencyAlert({
  token, userId, alertType, deviceId, message
}: { token: string; userId: string; alertType: string;
     deviceId: string; message: string }) {
  await messaging.send({
    token,
    notification: {
      title: '紧急告警',
      body: message,
    },
    data: { alertType, deviceId, urgency: 'P0' },
    android: {
      priority: 'high',
      notification: {
        sound: 'emergency_alert.mp3',  // 自定义铃声
        channelId: 'emergency_alerts',
      }
    },
    apns: {
      payload: {
        aps: { sound: 'emergency_alert.wav', badge: 1 }
      }
    }
  });
}

export async function sendMedReminder(userId, pillInfo) {
  // P2 级别: 仅 APP 内消息 + FCM 低优先级
  await messaging.sendEachForMulticast({
    tokens: await getUserTokens(userId),
    notification: { title: '吃药提醒', body: pillInfo },
    data: { urgency: 'P2' }
  });
}

细微信号：家庭照护者的"数字疲劳"拐点
2024年多项研究显示，长期接收照护类Push通知的子女中，约35%在6个月内出现"通知疲劳"——表现为忽略P1/P2级别通知、关闭非紧急推送、甚至完全退出APP。这与日本2000年代"介护疲劳"(介護疲れ)现象高度相似：当数字化工具将无形的照护压力显性化为持续的告警流时，照护者本身的心理健康风险显著上升。系统设计必须包含"通知降噪"和"照护者关怀"模块，否则APP的留存率将在第8-12个月出现断崖式下跌。

21.5 多学派交叉解读：家庭APP生态的经济学意涵

演化经济学 (Nelson & Winter)
家庭APP生态的演进遵循"惯例-选择-保留"路径：从早期的"单一告警推送"到当前的"多角色权限+智能分级"，反映了市场对"照护效率"的选择压力。关键机制是路径依赖——现有APP的数据模型一旦锁定为"子女中心主义"(子女拥有全部管理权)，后来者很难切换到"老人赋权"模式，因为用户习惯和数据库架构都已固化。类似日本1990年代从"家族企业传承"向"职业经理人制度"的路径依赖困境。
行为经济学 (Kahneman & Tversky)
告警通知系统设计面临典型的"可得性启发"偏差：P0级别的紧急告警因为情绪冲击力强而被过度关注，P2级别的日常提醒则被系统性低估。这导致用户注意力资源配置扭曲——最重要的长期健康趋势(周报)反而获得最少曝光。最优通知策略应基于损失框架而非概率框架：不是"心率异常概率3%"，而是"过去3天心率持续偏离基线1.8σ"，后者触发更强的行动动机。
信息经济学 (Akerlof & Stiglitz)
家庭APP生态中存在严重的信息不对称：子女(管理者)掌握全部数据，老人(被照护者)仅有有限可见性。这种不对称可能导致"逆向选择"——老人因隐私担忧而拒绝使用设备，最终导致高风险老人(最需要监护的人群)反而被排除在服务之外。解决之道在于信号传递机制：通过透明的数据使用协议和老人端数据可见性设计，向老人传递"隐私受保护"的信号，缓解逆向选择。
新制度经济学 (North & Williamson)
多角色权限体系本质上是一种治理结构创新。Williamson的交易成本理论指出：当资产专用性高(健康数据具有唯一性)、不确定性高(老人健康状况波动)、交易频率高(每日多次数据交互)时，最优治理结构是"混合制"——这正是家庭APP中"子女管理+医护协作+老人自主"的三级治理架构。中国《个人信息保护法》对健康数据处理的"单独同意"要求，实际上提高了家庭数据共享的制度性交易成本。
公共选择理论 (Buchanan & Tullock)
家庭APP中的"主照护者(admin)"角色设定隐含了一个政治学问题：谁有权代表老人做出决策？Buchanan的公共选择理论指出，任何决策权的集中都会产生"代理人问题"——admin角色可能在未经老人同意的情况下修改围栏范围、关闭告警通知、甚至导出健康数据给第三方。最优制度设计需要引入"制衡机制"：老人端保留"否决权"按钮，当admin操作涉及隐私边界时可触发重新协商。
案例深挖：平安好护士APP的家庭协作模式
背景：平安集团2022年推出的"好护士"APP，面向800万慢病管理用户，核心功能是"患者-家属-护士"三方协作。

设计亮点：(1) 引入"照护轮值表"功能——子女可在线排班，系统自动分配P1/P2通知接收人，避免所有子女同时收到相同通知；(2) "数据脱敏开关"——老人可自主决定哪些数据对哪些家属可见，例如血糖数据仅对内分泌科医生可见；(3) "告警疲劳指数"——系统实时计算每个子女本周收到的通知数量和类型分布，当P1+通知超过20条/周时自动触发"降噪建议"。

效果：上线6个月后，家属端APP月活从42%提升至67%，告警疲劳投诉下降58%。关键成功因素是将"照护者心理健康"纳入产品设计指标，而非仅关注"被照护者安全"。

经济学解读：平安的方案本质上是在家庭内部建立了"信息治理市场"——通过轮值机制降低单个代理人的信息过载，通过脱敏开关建立"数据产权"分配，通过疲劳指数引入"信息税"概念(过量通知被视为对注意力的征税)。

战略建议：家庭APP设计的三个关键决策点
第一，照护者疲劳是隐形杀手：不要只设计"通知发送"逻辑，还要设计"通知消费"逻辑。引入告警疲劳指数和自动降噪机制，将照护者的心理健康纳入产品KPI。

第二，数据产权归属是法律红线：健康数据的所有权在法律上属于老人，但实际控制权在子女手中。必须在产品设计层面实现"老人数据自主权"——至少让老人可以查看"谁在什么时候看了我的什么数据"。

第三，轮值机制是留存率的关键：当多个子女共享照护责任时，"谁来负责"是最大摩擦点。引入在线轮值表和通知分发规则，可以将家庭内部协调成本降低60%以上，这也是从"工具"升级为"平台"的关键一步。
