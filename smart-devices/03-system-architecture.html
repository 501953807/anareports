第三章：四层系统架构与数据流 — 从传感器到行动执行

系统分为感知层(手环/药盒采集数据)、传输层(Cat1/BLE/WiFi/微信)、平台层(云端服务+AI引擎)、应用层(老人/家属/运营交互)。数据从传感器到行动执行的完整闭环。

3.1 四层架构详解

层级组成技术关键指标
感知层手环(定位+健康传感器)、药盒(分药机构+检测)、家属手机、社区/医院GPS/北斗、PPG/SpO2、IMU、步进电机、红外检测、TTS语音定位精度<3m、跌倒检测F1>0.92、分药误差<1片
传输层Cat1蜂窝、BLE 5.0局域网、WiFi、4G/5G移动网络、微信小程序MQTT over TLS、WebSocket、HTTPs、BLE GATT手环持续在线、药盒家庭WiFi、APP移动网络、微信触达
平台层设备管理、消息队列(RabbitMQ/Kafka)、数据存储(MySQL+Redis+MongoDB)、AI引擎、推送服务、RESTful APIGo/Java后端、Docker+K8s、PyTorch(ML模型)跌倒检测端侧推理<30ms、告警P0级别优先路由
应用层老人端(语音提醒+LED+SOS)、家属端(APP/小程序)、运营端(Web后台)Swift(Kotlin)、Vue/React、高德/百度地图API多账号共享、健康周报生成、B2B客户管理

3.2 业务链闭环 — 6步数据流

步骤描述技术细节
Step 1老人佩戴手环、使用药盒手环持续采集：GPS位置(每30s) + 心率(每5min) + 加速度(100Hz采样)
Step 2数据通过Cat1/BLE传输位置+心率→Cat1蜂窝→云端；手环↔药盒→BLE 5.0局域网(功耗<1mA)
Step 3云端AI实时分析跌倒检测模型(M4端侧推理<30ms)→异常事件→标记P0级别→消息队列优先路由
Step 4多维推送告警APP推送(APNs/厂商通道)→微信订阅消息→SMS短信兜底→电话语音(紧急事件)
Step 5家属响应 + 药盒联动家属APP确认→电话老人→未接听自动呼叫社区/120；药盒根据手环健康数据调整用药提醒策略
Step 6数据沉淀 + 模型迭代所有事件→MongoDB时序存储→AI模型每周更新→跌倒检测准确率持续提升

3.3 关键技术参数

维度参数目标值
定位精度GPS+北斗+GLONASS室外<3米
基站辅助(WiFi)室内10-50米
告警时更新频率1秒/次(正常30秒/次)
健康监测心率精度±2bpm
跌倒检测F1分数>0.92，假阳性<1次/天
血氧精度(消费级)±2%
药盒性能分药精度单次误差<1片
可靠性>10,000次循环无故障
续航2500mAh，WiFi待机7-15天

实操：从零搭建智能穿戴系统架构 — 手把手教程

第一步：画架构图(Draw.io教程)

  1. 访问 draw.io (app.diagrams.net)，新建空白文档
  2. 左侧面板搜索"aws"加载AWS图标库，搜索"arrow"加载流程箭头
  3. 画4个矩形框作为四大层级，从上到下排列：感知层 → 传输层 → 平台层 → 应用层
  4. 在"感知层"内添加形状：圆形(手环图标) + 方形(药盒图标) + 手机图标 + 医院图标
  5. 用粗蓝色箭头标注数据流：手环 → Cat1 → 云端 → APP → 子女手机
  6. 导出为PNG(300dpi)插入PRD文档，命名为"SYS-ARCH-v1.png"

第三步：BLE GATT自定义服务设计

UUID类型方向用途数据格式
0000fff0-0000-1000-8000-00805f9b34fbService-自定义健康监测服务-
0000fff1-...-34fbCharacteristicDevice→APP心率数据2字节uint16(bpm)
0000fff2-...-34fbCharacteristicDevice→APP血氧数据1字节uint8(%)
0000fff3-...-34fbCharacteristicDevice→APP位置坐标4字节lat(int32) + 4字节lon(int32)
0000fff4-...-34fbCharacteristicAPP→Device发送用药提醒4字节时间戳(uint32)
0000fff5-...-34fbCharacteristicAPP→Device电子围栏设置经纬度数组[lat,lon,radius]
0000fff6-...-34fbCharacteristicDevice→APP电池电量1字节uint8(%)
BLE GATT编码技巧
经纬度使用int32存储(实际值×10,000)，例如116.4度存为1164000。这样可以压缩传输数据量。Heart Rate Service使用标准UUID 0x2A37，建议保留标准UUID提高兼容性。

第六步：阿里云部署操作清单

  1. ECS购买：选择ecs.c7.xlarge(4核8G)，CentOS 7.9，按量付费。首年约¥2,500。地域选杭州(靠近华东用户)。
  2. RDS购买：MySQL 8.0高可用版，2核4G，100GB SSD。首年约¥3,600。开启Binlog备份。
  3. Redis购买：1GB标准版，用于session存储。开启SSL。
  4. OSS购买：100GB存储+CDN流量包，用于APP安装包和健康报告PDF存储。
  5. 安全组配置：仅开放443(HTTPS)、22(SSH仅限VPN IP)、8883(MQTT TLS)端口。其余全部拒绝。
  6. Docker安装：curl -fsSL https://get.docker.com | sh && systemctl enable docker

第八步：系统监控与告警配置

  1. 安装Prometheus+Grafana：docker run -p 9090:9090 prom/prometheus:latest。访问localhost:9090/grafana配置Dashboard，监控CPU/内存/磁盘/网络流量。
  2. 配置告警规则：在Prometheus alert.rules文件中定义规则：if cpu_usage > 80% for 5min then alert("CPU过载")；if mysql_connections > 90% then alert("数据库连接池接近上限")。
  3. 接入钉钉/企业微信机器人：用Webhook把告警推送到团队群。关键告警(P0/P1)同时触发SMS短信，确保值班人员24小时响应。
  4. 日志收集：安装ELK(Elasticsearch+Logstash+Kibana)或使用阿里云SLS，统一收集所有微服务的日志。设置关键词告警：ERROR/FATAL/Exception。

第九步：系统压测方案(Load Testing)

测试场景并发用户数预期响应时间工具

系统架构设计核心原则
记住三条铁律：第一，定位和告警永远高优先级——这些数据不走消息队列排队，走独立Redis channel直接推。第二，所有时间戳统一用UTC存储，展示时转为东八区+夏令时偏移。第三，老人端设备掉线不崩溃——手环本地缓存3天的健康数据和轨迹，恢复连接后断点续传。
