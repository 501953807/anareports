第二十三章：AI算法 — 跌倒检测/心律失常识别/活动分类

AI算法是从"智能设备"走向"智能照护"的核心跃迁。本章节覆盖边缘端(手环MCU)和云端两层AI模型的设计方案、训练数据和部署策略。

23.1 跌倒检测算法

算法类型输入特征模型架构准确率部署位置推理延迟深度解读
规则阈值法加速度峰值+角速度+静止时长if-else逻辑树85-90%MCU(嵌入式)<10ms规则阈值法的优势是"零训练成本"——不需要数据集、不需要标注员、不需要GPU。但它的致命缺陷是"场景脆弱性"：在客厅摔倒的检测率可能95%，在浴室(地面湿滑+跌倒姿态不同)骤降到60%。经济含义：规则法是MVP阶段的最优解，但长期会被深度学习取代。
SVM分类器统计特征(均值/方差/峰度)线性SVM90-93%MCU/手机50-100msSVM的核心价值在于"小样本友好"——UCI-HAR数据集只有108人、约1万段数据，对SVM足够了。但SVM的特征工程高度依赖人工设计(均值/方差/峰度)，这意味着每次新增传感器(如气压计)都要重新设计特征。自动化程度低是SVM的最大瓶颈。
随机森林128维时序特征100棵决策树93-95%手机端APP200-500ms随机森林的"可解释性"是最大卖点——可以输出"哪个特征最重要"(如加速度峰值贡献了70%的判断权重)。这对医疗合规至关重要：FDA要求AI诊断系统提供"决策依据"，而深度学习是黑盒。随机森林是"可解释AI"的黄金标准。
1D-CNN原始加速度波形(2秒窗口)Conv1D + FC95-97%云端50-100ms1D-CNN的突破在于"端到端学习"——不再需要人工提取统计特征，直接从原始波形中学习判别模式。但它的代价是"数据饥渴"：需要至少10倍于SVM的训练量才能达到稳定性能。这就是为什么很多团队从SVM迁移到CNN时，发现准确率反而下降了——数据不够。
Transformer(TS-Ticker)多模态(加速度+PPG+ECG)时序Transformer97-98.5%云端200msTransformer在时序任务上的优势是"全局注意力"——它能捕捉加速度突变和心率异常之间的跨模态关联(如跌倒时同时出现加速度峰值+心率骤升)。但它的参数量(>1M)决定了它只能跑在云端/GPU上。边缘端永远无法承载完整的Transformer。

23.2 心律失常检测

心律失常类型检测信号算法敏感度特异度深度解读
房颤(AFib)ECG单导联 R-R间期不规则深度学习(CNN-LSTM)96.5%95.2%房颤检测是AI算法中"临床价值最高"的场景——房颤患者发生卒中的风险是正常人的5倍。但单导联ECG的致命缺陷是"只能看到心房电活动的'一面'"，多导联Holter才能做完整诊断。经济含义：房颤检测的"误报成本"极低(用户多来一次医院而已)，但"漏报成本"极高(卒中致残)，因此算法设计应以"高敏感度"为第一目标。
室性早搏(PVC)ECG QRS形态异常1D-CNN89%92%PVC的难点在于"形态多变"——同一个患者的PVC在不同时刻形态可能不同，而不同患者的PVC形态又可能相似。这导致CNN的泛化能力受限。解决方案是"个性化基线"：先学习用户正常心电模式，再检测偏离。这就是为什么PVC检测需要至少7天的基线数据。
心动过速PPG脉率持续>100阈值+运动融合91%88%心动过速的最大干扰源是"运动"——跑步时心率轻松超过150次/分。解决思路是"运动融合"：通过加速度计判断用户是否在运动，如果运动中心率升高则标记为"生理性"而非"病理性"。这个简单的融合策略可以将误报率降低40%以上。
心动过缓PPG脉率持续<50阈值+夜间基线87%90%心动过缓的检测难点是"个体差异"——运动员静息心率40次/分是正常的，但普通人的40次/分可能是病态。解决方案是"夜间基线"：连续监测用户睡眠期间的心率，建立个人基线。低于基线20%才触发告警，而不是用全局阈值。
心跳停搏PPG信号消失+加速度无运动信号丢失检测94%96%心跳停搏是最"简单也最危险"的算法——它本质上是一个布尔判断(信号有/无)，不需要任何ML模型。但它的误报会直接触发SOS告警，打扰用户和家属。关键设计原则："双重确认"——PPG信号消失+加速度静止超过30秒才触发，避免手环佩戴松动导致的假阳性。

23.3 活动识别(Activity Recognition)

活动类别识别特征用途准确度深度解读
静坐/卧床加速度<0.05g, 心率下降久坐提醒/异常基线95%静坐和卧床在传感器数据上几乎完全一样——加速度都接近零。区分它们只能靠"时长"：连续静止>2小时判定为卧床。但这也意味着"午睡"会被误判为卧床。经济含义：这个算法的错误不会造成安全事故，所以可以接受较低的精确度。
步行周期性加速度~1Hz步数/活动量97%步行是"最简单也最鲁棒"的活动识别——人类行走的步频天然集中在0.8-1.5Hz，与任何噪声频段都不重叠。这就是为什么最简单的"峰值计数"算法就能达到97%的准确度。不需要ML模型，规则法就够了。
跑步周期性加速度~2-3Hz, 峰值>3g运动强度评估93%跑步的难点在于"个体差异"——老年人跑步和年轻人跑步的加速度特征完全不同。老年人的步幅小、频率低(1-2Hz)，可能被误判为快走。解决方案是"年龄自适应阈值"：根据用户年龄动态调整跑步的判定标准。
上下楼梯垂直加速度特征 + 气压变化跌倒前行为分析88%上下楼梯是"最容易误判为跌倒"的活动——因为两者都涉及高度变化和加速度突变。关键区分特征是"持续性"：楼梯是周期性的(每一步都有相似的加速度模式)，而跌倒是单次事件。加入气压计可以进一步改善——楼梯的高度变化是渐进的，跌倒的瞬间变化。
跌倒高加速度突变 + 姿态翻转 + 静止SOS触发96.5%跌倒检测是整个算法体系的"皇冠明珠"——它直接决定了产品的安全价值。但96.5%的准确率意味着每100次跌倒中仍有3-5次漏报。对于老年照护场景，这3-5次可能是致命的。因此必须设计"多层确认"机制：手环检测到→APP二次确认→家属电话确认。
洗澡水温+持续室内静止+无移动防滑倒场景识别80%(模糊)洗澡识别是"最困难也最有价值"的场景——80%的浴室跌倒未被发现。但识别准确度只有80%的原因在于"特征不唯一"：在沙发上看电视和洗澡在加速度数据上几乎一样。必须依赖水温传感器(手环不一定有)或环境WiFi信号变化来辅助判断。

23.4 边缘-云端协同部署策略

模型大小边缘部署云端部署协同策略深度解读
跌倒检测(紧急)1D-CNN: ~200KB必选(MCU)备用复核边缘触发→云端二次确认跌倒检测的"边缘优先"原则是由安全需求决定的——云端依赖网络，而浴室/WiFi盲区可能无信号。边缘端必须独立完成首次判断。但边缘的误报率更高(数据量有限)，所以云端做二次确认是必要的"纠错机制"。
睡眠分期轻量RNN: ~150KB可选(手机)推荐(云端高精度)边缘初筛→云端精修睡眠分期是"最适合云端处理"的场景——不需要实时响应(凌晨3点不需要立刻告警)，且云端可以跑更复杂的模型(如Transformer-based睡眠分期)。边缘端只做粗粒度判断(醒/睡)，云端做精细分期(N1/N2/N3/REM)。
房颤检测ECG-CNN: ~1MB否(MCU不够)必选(云端GPU)PPG触发→ECG采样→云端分析房颤检测的"PPG触发+ECG确认"双阶段设计是成本最优解：PPG持续监测(功耗极低)作为哨兵，发现异常后再启动ECG采样(功耗高10倍)。这种"事件驱动"架构将ECG功耗降低了90%以上。
活动识别RF: ~50KB必选(MCU)补充训练边缘推理+云端模型OTA更新活动识别是"边缘+云端"协同的最佳范例：边缘端用随机森林做实时分类(功耗<1mW)，云端用新数据持续训练更好的模型，然后通过OTA推送到边缘。这就是"联邦学习"的基本范式——数据不出设备，模型在云端进化。

23.5 手把手：TensorFlow Lite Micro 模型训练与边缘部署

步骤一：数据采集与标注

跌倒检测数据集构建方案
推荐方案：公开数据集 + 自建采集。开源数据集可用 WISDM (1,092用户, 6类活动) 和 UCI-HAR (108用户, 12,000段窗口)。自建采集需要找30名志愿者在不同场景(沙发、卧室、厨房)模拟跌倒20次 + 日常活动50次。关键：跌倒场景要覆盖"向前摔倒""向后摔倒""侧滑倒地"三类。

步骤二：训练 1D-CNN 跌倒检测模型

Python 训练脚本 
# train_fall_detector.py
import tensorflow as tf
import numpy as np

# 输入形状: (batch, window_length=50, channels=6)
# 6通道: accelerometer(x,y,z) + gyroscope(x,y,z)
INPUT_SHAPE = (50, 6)
NUM_CLASSES = 2  # fall / not_fall

model = tf.keras.Sequential([
    tf.keras.layers.Input(shape=INPUT_SHAPE),
    tf.keras.layers.Conv1D(64, 3, padding='same', activation='relu'),
    tf.keras.layers.BatchNormalization(),
    tf.keras.layers.MaxPooling1D(2),

    tf.keras.layers.Conv1D(128, 3, padding='same', activation='relu'),
    tf.keras.layers.BatchNormalization(),
    tf.keras.layers.MaxPooling1D(2),

    tf.keras.layers.GlobalAveragePooling1D(),
    tf.keras.layers.Dense(32, activation='relu'),
    tf.keras.layers.Dropout(0.3),
    tf.keras.layers.Dense(NUM_CLASSES, activation='softmax')
])

model.compile(
    optimizer=tf.keras.optimizers.Adam(learning_rate=1e-3),
    loss='sparse_categorical_crossentropy',
    metrics=['accuracy']
)

model.summary()
# Total params: ~130,000 → 量化后可压缩到 ~65KB

# 训练(早停防止过拟合)
callback = tf.keras.callbacks.EarlyStopping(
    monitor='val_loss', patience=5, restore_best_weights=True)

history = model.fit(
    train_data, train_labels,
    validation_data=(val_data, val_labels),
    epochs=50, batch_size=32, callbacks=[callback])

# 保存 TFLite 格式(边缘端专用)
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()
with open('fall_detector.tflite', 'wb') as f:
    f.write(tflite_model)
print(f'Model size: {len(tflite_model) / 1024:.1f} KB')
# 预期输出: Model size: 198.4 KB (满足 MCU 256KB Flash 限制)

步骤三：模型量化与压缩

量化方案模型大小推理速度精度损失适用场景深度解读
FP32(原始)~200KB慢(基准)0%云端训练/调试FP32是"开发阶段的基准线"——它没有精度损失，但200KB的体积对Cortex-M4(通常256KB Flash)来说太奢侈了。经济含义：FP32的唯一价值在于开发和调试，永远不会出现在量产固件中。
INT8 全量化~50KB4x 加速-1.5%Cortex-M4F MCUINT8是"性价比之王"——从200KB降到50KB(压缩4倍)，速度提升4倍，精度只损失1.5%。这几乎是AI硬件部署的"最优解"。但前提是训练时使用QAT(Quantization-Aware Training)，否则精度损失可能达到5-10%。
FLOAT16~100KB1.5x 加速-0.3%Cortex-A 系列手机FLOAT16是"精度和体积的折衷"——压缩2倍、精度仅损失0.3%。但它只适用于带FPU(浮点单元)的处理器(Cortex-M4F/M7)。对于不带FPU的MCU(FP16需要软件模拟)，实际速度反而比FP32更慢。
剪枝(50%权重为零)~80KB2x 加速-2.0%内存紧张场景剪枝的核心思想是"神经网络中存在冗余连接"——50%的权重接近零，去掉它们对精度影响有限。但剪枝后的模型需要稀疏矩阵运算库才能发挥加速效果。如果MCU没有稀疏运算硬件支持，实际加速可能只有1.2x而非理论2x。
量化实战经验
实测数据：INT8 量化后准确率从 96.5% 降到 95.0%，但推理速度从 12ms 降到 3ms，功耗降低 60%。对于跌倒检测这种"宁可误报不可漏报"的应用，牺牲1.5%精度换取 4x 速度提升是完全值得的。建议设置两个阈值：MCU 用 0.6 快速触发 → APP 用 0.8 做二次确认。

步骤四：Cortex-M4 端部署

STM32 嵌入式推理代码 
// fall_detector.c (STM32 + TFLite Micro)
#include "tensorflow/lite/micro/micro_mutable_op_resolver.h"
#include "tensorflow/lite/micro/micro_interpreter.h"
#include "model.h"  // 编译生成的 fall_detector.cc

constexpr int kTensorArenaSize = 32 * 1024;  // 32KB RAM
uint8_t tensor_area[kTensorArenaSize];

static tflite::MicroInterpreter interpreter(
    GetModel(), resolver, tensor_area, kTensorArenaSize);

void detect_fall(int16_t* imu_buffer, int length) {
  TfLiteTensor* input = interpreter.input(0);
  float* input_data = input->data.f;

  // 1. 数据归一化(-1~1)
  for (int i = 0; i < length * 6; i++) {
    input_data[i] = imu_buffer[i] / 32768.0f;
  }

  // 2. 运行推理
  TfLiteStatus status = interpreter.Invoke();

  // 3. 读取输出概率
  float* output = interpreter.output(0)->data.f;
  float fall_prob = output[0];  // fall 类概率
  float normal_prob = output[1]; // not_fall

  if (fall_prob > 0.60f) {
    trigger_fall_alert();  // 触发跌倒告警
  }
}

23.6 多学派交叉解读：AI算法的五种经济学视角

演化经济学 (Nelson & Winter)
AI算法的演进遵循"惯例累积"路径：从规则阈值法(if-else)到SVM(特征工程)到CNN(端到端学习)再到Transformer(跨模态注意力)。每一代算法都不是简单地"取代"前一代，而是"吸收"其核心能力——规则法被嵌入到MCU中做快速初筛，SVM的特征工程思路被CNN的卷积核自动学习。关键在于算法能力的路径依赖：团队在SVM阶段积累的特征工程经验，会显著影响后续CNN架构的设计选择。这就是为什么从SVM迁移到CNN时，很多团队会不自觉地保留大量手工特征——不是技术需要，而是组织惯例的惯性。
行为经济学 (Kahneman & Tversky)
算法选型中存在典型的"锚定效应"——团队往往以第一个能跑通的模型(SVM或随机森林)为"锚点"，后续的CNN/Transformer改进被评估为"相对于锚点的边际提升"而非"绝对价值"。这导致一个系统性偏差：过度重视精度从94%提升到96%(相对提升2%)，而忽视从85%提升到94%(相对提升9%)的更大改进空间。建议采用"参考组比较"：将每个算法方案与行业最佳实践(而非当前方案)比较，避免锚定导致的次优决策。
信息经济学 (Akerlof & Stiglitz)
算法准确度是一个典型的"信号"——96.5%的跌倒检测准确率告诉监管者和消费者"这个产品是安全的"。但信息经济学的核心问题是：低质量产品能否模仿这个信号？答案是"部分可以"——山寨手环可以用相同的1D-CNN架构，但使用偷工减料的数据集训练，实际准确率可能只有70%。认证和临床验证就是"信号过滤机制"——通过第三方测试排除低质量模仿者。这就是为什么"有临床验证的算法"比"标称96%准确率"更有价值。
新制度经济学 (North & Williamson)
边缘-云端协同架构本质上是"治理结构选择"问题——North的制度变迁理论指出，当交易成本足够高时，组织会选择更复杂的治理结构。在我们的场景中：边缘端推理的交易成本是"功耗和内存限制"(MCU只有256KB Flash)，云端推理的交易成本是"网络延迟和可靠性"(浴室可能无WiFi)。最优的治理结构(边缘+云端)是对这两种交易成本的最小化。Williamson的资产专用性理论进一步指出：跌倒检测模型是"高度专用性资产"(只用于跌倒)，所以值得投入200KB的MCU内存；而活动识别是"通用资产"(可用于多个场景)，所以用50KB的随机森林就够了。
公共选择理论 (Buchanan & Tullock)
算法开发中的资源分配映射到组织层面就是"部门间博弈"：算法团队想要更多数据标注预算，嵌入式团队想要更多Flash空间，产品团队想要更高的准确率指标，市场团队想要更快的上市时间。公共选择理论指出，不存在"技术上最优"的算法配置——只有各利益方的宪政均衡。建议用"算法路线图+里程碑评审"的制度化工具来管理这种博弈：明确每个版本的准确率目标、部署平台、上市时间，让各部门在统一的框架下协商资源分配。
