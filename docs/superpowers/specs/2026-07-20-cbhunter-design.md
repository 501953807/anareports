## 四、系统架构设计

### 4.1 多平台发布功能架构：混合架构（推荐）

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A) 统一底座 + 平台扩展 | 一套核心代码，各平台插件式扩展 | 维护成本低、字段映射清晰 | 平台差异大会导致底座臃肿 |
| B) 独立发布器 | 每个平台单独开发一套 | 灵活性高、互不影响 | 重复开发、维护成本高 |
| C) 混合架构 | 核心业务层统一 + 平台适配层独立 | 兼顾维护性和灵活性 | 需要清晰的接口定义 |

**最终选择: C) 混合架构**

理由：
1. 商品/订单/资金是通用业务，应该统一
2. 但各平台的分类体系、SKU 规则、定价逻辑差异太大，硬塞进统一底座会很难维护
3. 妙手 ERP 也是类似思路——核心功能统一，平台适配独立

### 4.2 店铺 - 产品层级结构

```
平台 (TikTok/Shopee/Temu)
├── 店铺 (菲律宾/新加坡/马来...)
│   ├── 商品列表 (树状视图)
│   │   ├── 商品 A
│   │   │   └── SKU 1 / SKU 2 / SKU 3
│   │   └── 商品 B
│   └── 营销活动
│       └── 限时折扣 / 优惠券 / 满减
└── 订单管理
    └── 待发货 / 已发货 / 售后
```

### 4.3 国别差异化解决方案

**数据层:**
- 统一字段存主表，国别差异字段存扩展表
- `product_standard_fields` + `product_platform_extensions`

**业务层:**
- 每个平台有一个适配器（Adapter），负责：
  - 分类树动态加载
  - SKU 规则校验
  - 定价模板转换
  - 费用计算

**前端渲染:**
- 基础表单用统一组件
- 平台特有字段用动态表单引擎渲染
- 国别差异通过配置驱动，不需要硬编码

## 五、调研执行方法论

### 5.1 统一模板

每个平台按以下结构提取：
- 页面结构：侧边栏导航、主内容区布局、弹窗/抽屉
- 字段清单：所有输入框、下拉选项、日期选择器、复选框
- 操作流程：从入口到完成的标准路径
- 国别差异：各站点特殊配置项

### 5.2 执行顺序

Shopee（优先）→ TikTok Shop → Temu → 妙手 ERP

### 5.3 浏览器操作规范

- 仅使用 chrome-devtools MCP
- 全程只读，不点击保存/上架/发布等写操作
- 若端口断开，停止调研并告知用户
